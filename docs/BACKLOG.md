# Backlog: what is left

A consolidated, ranked view of the remaining work for memo-words: ideas (new
features), suggestions (improvements), and problems (defects/weaknesses) that are
**not yet done**. It complements the two other docs rather than replacing them:
[`RECOMMENDATION.md`](RECOMMENDATION.md) is the critical audit of existing code,
and [`DESIGN_IDEAS.md`](DESIGN_IDEAS.md) is the design-idea backlog. This file is
the single "what's next" list across all three lenses.

## How it was produced

A 16-lens generation pass (learning, content, UX, visual, accessibility,
correctness, architecture, tests, performance, build/release, platform, config,
observability, docs, integrations, security) produced **254 raw items**, which
de-duplicated to **139 genuinely distinct** remaining items. It was deliberately
**not padded to 200** - 115 raw items were real duplicates (persistence alone
collapsed 25 candidates into 9; testing 17 into 10; config/reload 13 into 9). An
honest 139 beats a padded 200. Already-shipped work (the 16 config keys, card
rendering, the type hierarchy, spaced recap, the modular refactor, the 66 tests,
the docs) is excluded.

## Shape of the work

- **By type:** 62 ideas, 51 suggestions, 26 problems.
- **By gate:** ~35 items are gated on infrastructure that does not exist yet
  (a persistence layer, cross-platform adapters); ~30 are testing/CI quality
  gates that unblock other work; the rest are doable now.
- **Roughly** 38% high-impact/high-effort (persistence, SRS, release), 42%
  medium (accessibility, config, integrations, docs), 20% polish.

## The single biggest lever

Almost everything that turns memo-words from a passive display into a real
learning tool sits behind **one** foundation: a **persistence layer** (ranks 1-2).
With a `Store` trait + a file/SQLite store, the gated cluster opens up - SM-2/FSRS
scheduling (#4), mastery levels (#5), history-aware recap (#6), streaks and daily
goals (#7), leech detection (#9), and tray-driven recall grading (#10). The cheap
enabler is making `WordSelector::choose()` take `&mut self` (#3, effort S) so a
scheduler can update per-word state. The other no-infrastructure cluster worth
doing early is **deck-data canonicalization** (#11-15): one versioned source of
truth that generates both the Mongo seed and the Rust fallback, ending the
seed/fallback drift and the duplicate/rank problems.

## Legend

- **Type** - `idea` (new capability), `suggestion` (improve existing),
  `problem` (defect/weakness).
- **Eff** - rough effort: S / M / L. **Imp** - impact: high / med / low.
- **Gate** - what it depends on: `persistence` (needs the state layer first),
  `cross-platform` (needs per-OS adapters), `breaks-clickthrough` (would require
  changing the no-in-card-input identity), or `-` (doable now).
